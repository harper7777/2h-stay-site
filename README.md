# 2H 分享家民宿網站 使用者需求規格書

## 一、專案名稱

2H 分享家民宿網站

## 二、專案目標

本專案旨在製作一個日月潭療癒系民宿網站，透過首頁品牌形象、房型介紹、線上訂房流程與交通資訊，讓旅客能在網站中快速理解民宿特色、比較房型，並送出訂房需求。

目前網站以「前端模擬訂房流程」為主要範圍，送出訂房需求後，可作為未來串接資料庫、後台管理與 CRM 的基礎。

## 三、網站定位

2H 分享家是一間主打療癒感、自然光、木質調與安靜住宿體驗的日月潭民宿。網站設計需傳達「慢下來、放鬆、療癒、自然」的品牌感受。

### 核心訴求

- 讓旅客快速理解民宿特色
- 讓旅客能瀏覽不同房型
- 讓旅客能依日期、人數查詢適合房型
- 讓旅客能填寫訂房資料並送出訂房需求
- 讓網站未來可延伸至資料庫、後台管理與顧客關係管理

## 四、使用者角色

### 1. 一般訪客

尚未決定是否訂房，只是進入網站瀏覽資訊的旅客。

### 2. 訂房旅客

已經有住宿需求，想查詢房型、比較房間，並送出訂房需求的旅客。

### 3. 民宿方

負責確認旅客訂房需求、回覆房況、安排入住與維護房型資訊的人員。

### 4. 未來管理者

未來若擴充後台，可管理房型、訂單、旅客資料與網站內容的人員。

# 五、使用者故事 User Story

## 一般訪客

1. 我想瀏覽首頁：快速了解 2H 分享家的品牌氛圍與網站主要內容。
2. 我想了解民宿特色：確認這間民宿是否符合我想要的住宿風格。
3. 我想閱讀品牌故事：了解民宿為什麼主打療癒、自然與慢生活。
4. 我想查看房型總覽：快速比較有哪些房型可以選擇。
5. 我想查看房型詳細資訊：了解每間房的入住人數、設備、空間特色與適合對象。
6. 我想查看交通資訊：確認民宿位置、地址與周邊交通方式。
7. 我想查看周邊景點：判斷這間民宿是否適合安排日月潭或九族文化村行程。
8. 我想快速前往訂房頁：在看完房型後能立即進入訂房流程。

## 訂房旅客

1. 我想輸入入住日期與退房日期：讓系統知道我的住宿期間。
2. 我想輸入入住人數：讓系統依人數顯示適合的房型。
3. 我想查詢可預約房型：快速知道目前有哪些房型符合條件。
4. 我想選擇房型：根據人數、風格與設備挑選想入住的房間。
5. 我想查看訂房摘要：確認房型、日期、人數與初步資訊是否正確。
6. 我想填寫訂房資料：留下姓名、電話、Email 與方便聯絡時段。
7. 我想填寫備註：補充特殊需求，例如入住時間、同行者需求或其他詢問。
8. 我想送出訂房需求：讓民宿方收到我的預約資料。
9. 我想看到送出成功提示：確認我的訂房需求已成功送出。
10. 我想知道訂房仍需民宿方確認：避免誤以為送出表單後就代表完成付款或保留房間。

## 民宿方

1. 我想維護房型資訊：更新房型名稱、圖片、介紹、入住人數與設備。
2. 我想確認旅客送出的訂房需求：查看日期、人數、房型與聯絡資料。
3. 我想確認房況是否可安排：判斷旅客選擇的日期與房型是否能入住。
4. 我想回覆旅客訂房結果：通知旅客是否可入住或需要改期、改房型。
5. 我想提供入住注意事項：讓旅客知道入住時間、退房時間與住宿規範。
6. 我想整理訂房紀錄：方便後續管理旅客資料與訂單狀態。

## 未來管理者

1. 我想管理首頁內容：更新網站主視覺、標語與品牌介紹。
2. 我想管理品牌故事內容：更新「水。樹」、「視。室」等故事區塊。
3. 我想管理房型資料：新增、修改或下架房型。
4. 我想管理房型圖片：上傳、刪除或調整房間照片排序。
5. 我想管理訂房需求：查看所有旅客送出的訂房表單。
6. 我想管理訂單狀態：標記訂單為待確認、已確認、已取消或已完成。
7. 我想管理交通資訊：更新地址、Google 地圖連結與交通說明。
8. 我想匯出訂房資料：方便整理報表或後續客服聯繫。

# 九、非功能需求 Non-functional Requirements

## 1. RWD 響應式設計

網站應支援桌機、平板與手機瀏覽，並確保 Navbar、房型卡片、訂房表單在小螢幕上仍能正常閱讀與操作。

## 2. 可讀性

網站文字應保持足夠字級、行距與留白，使使用者能快速理解資訊。

## 3. 一致性

所有頁面應維持一致的視覺風格，包含：

- 色系
- 字體
- Navbar
- Footer
- 按鈕樣式
- 卡片樣式

## 4. 易用性

主要功能入口應清楚可見，例如：

- 查看房型
- 立即訂房
- 查詢空房
- 送出訂房需求
- 查看交通資訊

## 5. 表單安全性與正確性

訂房表單需進行基本欄位檢查，避免使用者送出空白或格式錯誤資料。

## 6. 可擴充性

目前為前端模擬流程，但架構需保留未來串接後端資料庫、後台管理與 CRM 的可能性。
